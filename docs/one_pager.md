# The Saboteur - One-Page Project Description

## The 2026 problem

AI tools now give _correct_ answers most of the time. That's the danger, not the cure. Research shows over half of STEM students rely on intuition rather than systematic checks to detect AI errors, and a confident fluent answer trains them to stop checking. The failure mode is **over-trust**: the AI sets up a problem correctly, makes a logic slip midway, and carries it forward to a confident final answer without self-correction. A better model makes this _worse_, so a judge can't dismiss it with "just use a better model."

Existing "AI literacy" tools are curricula and PDFs - _reading material_, not interactive practice. There is no consumer tool that systematically trains a student to catch a wrong AI answer, on calibrated mistakes, with feedback that measures whether they're improving.

## What we built

**The Saboteur** is a single-player game that flips the role: every other tool positions the AI as authority answering the student; we make the student the authority _auditing_ the AI. The player sees a step-by-step worked algebra solution that is either fully correct or contains exactly one planted error. They Trust or Flag the broken step. The system grades instantly against hidden ground truth and shows the misconception they missed (or caught).

## The architectural moat

**SymPy decides correctness. The LLM only narrates.** Every problem is generated by a symbolic engine; errors are injected at known step indices using a misconception library of 12 transforms, each unit-tested to produce verifiably-wrong but well-formed output. The grader compares against the hidden truth label - no LLM involvement. The LLM's only roles are phrasing each step in natural language and explaining the misconception _after_ the reveal, both behind prompts that forbid evaluative correctness claims.

This is why the project can't be "confidently wrong." A judge asking "what if the LLM hallucinates?" gets a one-sentence answer: it can't affect grading by construction.

## The calibration metric

Score isn't raw accuracy - it's a 2×2 of (Trust/Flag) × (Actually correct/Actually wrong). Over-trust is weighted 3× over-suspicion, directly encoding the research finding that fluent-but-wrong AI output trains the wrong habit. The session calibration score climbs as players improve, and the curve is real, verifier-backed, not a proxy.

## Architecture

```
Problem Generator (SymPy)
   ↓ canonical solution + step ops
Sabotage Engine                            Misconception Library
   ↓ inject one error at known step  ←──   (12 transforms, unit-tested)
Presentation (LLM - narrate only)
   ↓ rendered round (steps + LaTeX)
Frontend (React + KaTeX)
   ↑ player action
Grader (deterministic, never LLM)
   ↓ 2×2 outcome
Calibration Engine (Elo-based difficulty tuner)
```

## Phase 3 differentiator: Audit Real AI Answers

Paste any LLM's step-by-step algebra solution into the BYOAI tab. SymPy verifies each step against the previous one and tells you where the AI broke. This turns the game into a daily tool - for actual homework, actual conversations with ChatGPT.

## Scope discipline

**Deep tool, narrow domain.** Algebra only: linear (1-step, distribution, fraction) and quadratic (factoring, formula, perfect square). Six generators, 12 misconceptions, calibrated by pilot testing. We don't claim universal STEM coverage; we claim the domain we cover is verified deterministically and the misconceptions are based on documented student/AI failure modes.

## What separates us

- **Concept, not coat of paint.** The role-flip is the project. No AI-tutor wrapper.
- **Real technical depth.** Symbolic generation, misconception-based injection, deterministic grading. Not a wrapper around chat.
- **Honest metric.** The calibration score is verifier-backed across multi-session data.
- **Robustness by construction.** The architecture rule (verifier decides, LLM explains) means the system _can't_ be confidently wrong about correctness.
